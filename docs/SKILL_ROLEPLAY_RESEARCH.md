# 角色扮演类 Skills 调研：前任.skill / 同事.skill 的做法与可复用模式

本文记录对近期在 GitHub 上流行的“角色扮演类 skill 仓库”的调研结论：它们如何把**聊天记录/文档/照片等原材料**蒸馏成**可长期演化的 Persona + Memory/Work**，从而让上层 AI 在“像某个人”这件事上更稳定、更真实。

> 说明：这些仓库面向的是“Claude Code Skills”的运行时，但其方法论（数据→蒸馏→运行→纠偏→版本）与具体平台无关，可直接迁移到任何上层 AI 应用。

---

## 1. 调研对象（样本）

本次重点阅读了以下仓库（均以 `SKILL.md` + `prompts/` + `tools/` 组织）：

- `perkfly/ex-skill`：前任.skill（WeChat/iMessage/SMS/照片/社交媒体 → Memories + Persona，支持增量更新与纠偏）
- `titanwings/colleague-skill`：同事.skill（飞书/钉钉/邮件/文档 → Work Skill + Persona，支持自动采集、增量更新与纠偏）
- `therealXiaomanChu/ex-skill`：前任.skill 的变体（增加 session summary 持久化、/scene 多人同场等能力）

---

## 2. 它们为什么“像人”：核心不是模型，而是“蒸馏管线”

这些 skill 的共同点是：**不直接把整段聊天扔给 AI**，而是做一条清晰的蒸馏管线，把原材料变成“可复用、可维护、可回滚”的结构化资产：

1) **最小化人工输入（3 问 intake）**  
只问 3 个问题拿到“锚点”（昵称/身份/标签），其它都可跳过；并且约定**手动标签优先级最高**，用于补齐原材料缺失、纠正模型误判。

2) **原材料导入（多来源）**  
聊天记录（WeChat/QQ/iMessage/SMS）、工作资料（飞书/钉钉/邮件/Wiki/PDF）、照片（EXIF 时间线）、社交媒体（导出/截图）。

3) **规范化（Parser → 中间表示）**  
用脚本把“各家导出格式”统一成一份**更适合 LLM 的摘要型文本**（Markdown/列表/统计特征 + 样本），而不是原封不动的大文件。

4) **双轨分析（Analyzer Prompts）**  
把抽取任务拆成两条“目标明确”的轨道：
- **Persona（怎么说、怎么想、遇到情况怎么做）**
- **Memory/Work（发生过什么、偏好是什么、能力与流程是什么）**

5) **构建（Builder Templates）**  
把分析结果写入稳定模板（分层 Persona、结构化 Memory/Work），并在末尾加上“运行规则”（先 Persona 决策，再 Memory/Work 补充，最后用 Persona 风格输出）。

6) **持续演化（Merge + Correction + Versioning）**  
增量追加、冲突提示、纠偏覆盖、版本存档/回滚，保证“越用越像”而不是“越聊越跑偏”。

## 3. 原材料如何变成可用输入：解析与压缩（Normalization）

### 3.1 聊天记录不是全文喂：先过滤、再分层采样

典型做法（以 WeChat 解析脚本为例）：

- **只抽取目标人物的消息**（用于提取口头禅、语气、表达习惯）  
  上层也可以同时保留“双方消息”用于冲突/修复模式，但“像谁”这件事优先看 ta 自己怎么说。
- **过滤系统占位符/媒体占位符**：如 `[图片]`、`[语音]`、撤回提示等（避免污染高频词统计）。
- **分类压缩**（把 token 花在最有信息密度的地方）：
  - 长消息（心情/观点/解释，权重高）
  - 情感关键词消息（争吵/想念/道歉等）
  - 日常沟通样本（用于风格参照，通常截取前 N 条）
- **提取统计特征**（变体实现更明显）：语气词、emoji、标点习惯、平均消息长度等。

这一步的本质是：**把“海量原始日志”压缩成“风格与事实足够、但 token 可控”的输入**。

### 3.2 其他来源的共同策略：先变时间线/要点，再喂给分析器

- 照片：先提取 EXIF 时间/地点，生成“时间线”；具体照片内容再按需打开。
- 社交媒体：优先抽取“第一人称表达 + 观点 + 情绪 + 生活片段”，避免无关互动噪音。
- 工作资料（同事.skill）：先抽取“负责范围/流程/规范/输出偏好/经验结论”，再按角色（后端/前端/PM/设计/数据）做专项维度。

## 4. Persona 设计：分层 + “行为规则化”是关键

这些仓库的 Persona 模板几乎都采用 5 层结构（表达略有差异，但目的相同）：

- **Layer 0：核心性格/硬规则（最高优先级）**
  - 强调：**不能写形容词，必须写“在什么情况下 → 会怎么做/怎么说”**  
    例：不是“ta很敏感”，而是“对方回得慢时会反复确认/阴阳怪气/冷处理”这种可执行规则。
  - 手动标签（如“爱撒娇/冷暴力/字节范”）会被翻译成 Layer 0 行为规则；并约定**手动标签 > 文件抽取**，冲突时明确标注。
- **Layer 1：身份**：职业/关系背景/MBTI 等（用于约束知识与视角）。
- **Layer 2：表达风格**：口头禅/高频词/句式/emoji/标点/消息密度，并给出“直接可用的话术例子”。
- **Layer 3：逻辑层**
  - 前任.skill：情感优先级、触发点、表达不满/退缩/修复方式等。
  - 同事.skill：决策优先级、推进/回避触发、如何拒绝/回应质疑等。
- **Layer 4：人际/关系行为**：对伴侣/朋友/家人/上级/平级/下级在不同压力下的表现。
- **Layer 5：边界与雷区**：明确不聊/不做/拒绝方式/底线。
- **Correction 记录**：用户一句“这不像 ta”触发纠偏，写成标准条目，长期覆盖运行（见第 6 节）。

为什么这套有效：  
它把“像一个人”拆成**可约束、可执行、可覆盖**的规则与风格，并把最关键的“不会做什么/不会说什么”前置成硬规则，降低漂移。

## 5. Memory / Work 的设计：事实与能力要“可检索、可引用、可过期”

### 5.1 前任.skill：Relationship Memory 的抽取维度

常见维度包括：

- 关系时间线：重要日期、里程碑、转折点
- 共同日常与仪式：固定活动、inside jokes、专属称呼/暗号
- 偏好：吃/玩/送礼与仪式感
- 冲突与修复模式：导火索、升级方式、冷战特点、和好信号、未解问题
- 情感动态：开心/难过/生气/想念时的信号与话术

并强调两点：
- **只写有依据的内容**；不足则标注“原材料不足”
- 记忆描述要具体、有画面感，避免“经常/可能/大概”

### 5.2 同事.skill：Work Skill 的抽取维度

Work 更偏“可执行能力”：

- 负责范围：系统/模块/术语/职责边界
- 工作流程：接任务、写方案、推进、处理异常
- 输出偏好：文档结构、结论前置、分点习惯
- 经验知识库：踩坑/观点/结论（最好可引用原话）

这让 AI 不只是“说话像”，还能“做事像”（写代码/写方案/CR/排障）。

## 6. 持续演化：增量合并、纠偏覆盖、版本回滚

角色扮演类应用最常见的失败原因是**漂移**与**误写入**。这些仓库用三件事控制质量：

### 6.1 增量 Merge：只追加，不覆盖；冲突要显式交给用户

增量更新时的规则通常是：

- 新内容若是补充细节 → 追加到对应章节
- 新内容若只是重复确认 → 忽略
- 新内容若与现有矛盾 → 输出“冲突提示”（列出旧/新/来源），让用户选择保留/替换/并存并标注时间

这种“append-only + 冲突提示”是可维护的关键：避免模型自作主张改写历史。

### 6.2 Correction：把“用户纠正”写成可执行规则，并设最高优先级

当用户说“这不对 / ta不会这样 / 感觉不像”时：

1) 提取：场景、错误行为、正确行为（模糊就追问一次确认）
2) 判断归属：属于 Persona 还是 Memory/Work
3) 生成标准条目：`- [场景: ...] 不应该... 应该...`
4) 写入到文件的 Correction 层，并作为运行时优先级最高的覆盖规则

这相当于给用户一个“训练入口”，用自然语言把模型拉回真实分布。

### 6.3 Versioning：每次更新先存档，支持 rollback

仓库通常会：

- 把 `memories.md / persona.md / SKILL.md / meta.json` 存入 `versions/vN/`
- `meta.json` 记录 `version / updated_at / corrections_count`
- 提供 `list / backup / rollback` 的命令行工具

这让上层产品可以实现：错误回滚、对比差异、审计变更。

## 7. “记忆不中断”的技巧：Session Summary（变体实现）

`therealXiaomanChu/ex-skill` 的一个很实用的点是：把“每次对话结束后的摘要”持久化，并在下次对话开场注入最近 N 条摘要作为上下文。

关键经验：

- 摘要结构固定：聊了什么/情绪基调/关键记忆点/下次可接着聊
- 加载时强调“自然延续”，不要变成汇报式复读
- 它比“把所有历史对话都塞进上下文”更可控，且更符合长期使用

对上层应用而言，这就是一个轻量的**长期对话记忆层**（不一定要向量检索也能跑起来）。

---

## 8. 运行时组装（Runtime Prompting）：先决策，再补充，再输出

这些仓库普遍在最终的 runnable `SKILL.md` 里写明运行规则（核心顺序一致）：

1) **Persona 先决策**：该用什么态度？会不会回？会怎么回应这个话题？
2) **Memory/Work 再补充**：把相关的共同记忆/能力规范拿出来
3) **用 Persona 风格输出**：口头禅、句式、emoji、标点习惯要一致
4) **优先级**：Layer 0（硬规则）/Correction 永远优先

这个顺序本质上是在做“Planner → Retriever → Generator”的最小实现。

## 9. 对我们上层产品的启示（结合本仓库定位）

本仓库是微信数据底座（不做 AI）。上层如果要做“像某个人”或“画像/记忆/个性化”，可以直接复用这些 skill 的方法论，但要注意工程落点：

1) **把“蒸馏产物”当资产管理**  
不要只存一段大文本。建议把 persona/memory 拆成结构化条目 + 版本 + 纠偏记录。  
本仓库已有建议模型：见 `docs/PERSONA_MEMORY_MODEL.md`（memories/runs/evidence）。

2) **让用户纠偏成为主流程**  
上层 UI 要提供“这不像 ta / 这条记忆不对 / 标记为待确认”的入口；并写入可追溯的 evidence。

3) **引入 run 元数据（即使 AI 不在底座）**  
上层每次调用 AI 生成/更新，都应该写回 runs：时间、模型、输入范围、成功失败原因（本仓库已有 runs 存取接口建议）。

4) **数据侧要提供“可蒸馏输入”而不是“全量倾倒”**  
上层蒸馏时最需要的通常是：
  - 单联系人的时间窗消息（近 7 天/近 1000 条/争吵相关片段）
  - 目标人物说话样本（用于风格）
  - 冲突与修复片段（用于 Layer 0/3）
  - 证据可跳转（local_id/timestamp/snippet）

5) **从轻量方案开始**  
先做“session summary + 结构化记忆条目”，通常就能显著提升连贯性；等需求更复杂再上向量检索/长期记忆压缩。

## 10. 常见坑（做产品时要提前防）

- **漂移**：没有 Layer 0/Correction 的硬约束时，模型会向“更讨好用户”的方向漂移，越来越不像。
- **误写入**：增量更新若允许覆盖，会把错误写死；应当 append-only + 冲突提示 + 回滚。
- **token 失控**：不做 parser/压缩，聊天记录越多越不可用；必须把“风格信息”和“事实信息”拆开压缩。
- **证据缺失**：没有 evidence（可回跳的消息片段），用户很难信任画像/记忆，也无法快速纠错。
- **安全边界**：前任类产品尤其要加边界（不鼓励骚扰/跟踪/替代真实沟通），并提供删除与导出。
